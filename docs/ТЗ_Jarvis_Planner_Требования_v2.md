## 1. Google Calendar — полная интеграция в режиме «лайв»

**Требование:** Как только встреча появляется в Google Calendar или в календаре на iPhone, она должна сразу отображаться в приложении.

### Реализация

| Уровень | Что сделать |
|--------|--------------|
| **Бэкенд** | OAuth 2.0 с Google (уже есть `credentials.json`). Эндпоинты: `GET /auth/google`, `GET /auth/callback`, `GET /auth/status`. Хранение токенов (refresh_token) по пользователю (сессия/привязка к устройству). |
| **Google Calendar API** | После авторизации — вызов `calendar.events().list()` с нужными календарями. Поддержка нескольких календарей (в т.ч. основной). |
| **Режим «лайв»** | **Вариант A:** Google Calendar API Push-уведомления (webhook): зарегистрировать канал на `https://backend/watch` с `address=jarvis-app.mooo.com`, при изменении событий Google шлёт запрос на бэкенд — бэкенд обновляет кэш и отдаёт клиенту при следующем запросе или через WebSocket/SSE. **Вариант B:** Периодический опрос с бэкенда (каждые 1–2 мин) и отдача клиенту единого списка. **Вариант C:** Клиент по таймеру или при открытии вкладки «Календарь» запрашивает у бэкенда актуальный список; бэкенд каждый раз ходит в Google Calendar API. Для «как только появилось» оптимально A или частый опрос (B/C). |
| **iPhone Calendar** | События из нативного календаря iOS — через **EventKit**: на клиенте запросить доступ к `EKEventStore`, читать события из выбранных календарей и показывать их вместе с событиями из Google (объединённый список в UI с пометкой источника). Либо бэкенд не участвует для iOS-календаря — только клиент объединяет два источника. |

### Итог

- Бэкенд: OAuth + Google Calendar API, эндпоинт вида `GET /calendar/events` возвращает события из Google (и при необходимости синхронизирует по webhook/таймеру).
- Клиент: при открытии Календаря и по pull-to-refresh запрашивает `GET /calendar/events`; при использовании EventKit — дополнительно запрашивает события из локального календаря и объединяет с ответом бэкенда в одном списке.

## 2. Gmail — полный доступ (чтение, написание, удаление)

**Требование:** Полный доступ к Google-почте: просмотр, написание писем, удаление и т.д.

### Реализация

| Уровень | Что сделать |
|--------|--------------|
| **Бэкенд** | Те же OAuth-токены Google (scope добавить: `https://www.googleapis.com/auth/gmail.readonly`, `gmail.modify`, `gmail.compose` и т.д. по необходимости). Эндпоинты: `GET /mail/messages` (список), `GET /mail/messages/{id}` (тело письма), `POST /mail/messages/send` (отправка), `DELETE /mail/messages/{id}` (удаление в корзину или полное), `POST /mail/messages/{id}/modify` (labels, mark read) при необходимости. |
| **Gmail API** | На бэкенде после авторизации: `users().messages().list()`, `get()`, `send()`, `trash()` / `delete()`. Формат ответов — JSON; для тела письма — декодирование base64url и парсинг MIME. |
| **Клиент** | Экран «Почта»: список писем (уже есть), тап — экран просмотра письма (тело, вложения по желанию). Кнопки: «Ответить», «Удалить». Экран «Новое письмо»: поля Кому, Тема, Текст; отправка через `POST /mail/messages/send`. В списке — свайп для удаления (вызов DELETE на бэкенде). |

### Итог

- Бэкенд: эндпоинты Gmail API (list, get, send, delete).
- Клиент: список писем, просмотр письма, написание нового, удаление (свайп и кнопка).

## 3. Фото блюда → нейросеть для калорийности

**Требование:** Пользователь снимает фото в приложении — нейросеть сразу даёт ответ по калорийности блюда.

### Реализация

| Уровень | Что сделать |
|--------|--------------|
| **Вид (Vision) API** | Использовать модель с распознаванием изображений. **Google Cloud Vision API** (метка объектов/еды) или **Gemini** (multimodal: изображение + промпт «Опиши блюдо и оцени калорийность») — дают текст о содержимом и можно вывести калории. Альтернатива: специализированные API для питания (если есть). |
| **Бэкенд** | Эндпоинт `POST /analyze-meal` уже есть. Заменить эвристику по размеру файла на: 1) принять image bytes; 2) отправить в Gemini (или Vision) с промптом вида «What dish is this? Estimate calories for a typical portion»; 3) распарсить ответ (или структурированный JSON от Gemini) и вернуть `{ "title": "...", "calories": N }`. Требуется API-ключ Google AI (Gemini) или сервисный аккаунт Google Cloud. |
| **Клиент** | Оставить текущий сценарий: выбор/съёмка фото → отправка на `POST /analyze-meal` → подстановка названия и калорий в форму приёма пищи. Дополнительно можно добавить кнопку «Сфотографировать» (камера) рядом с выбором из галереи. |

### Итог

- Бэкенд: интеграция с Gemini (или Vision) в `analyze-meal`; возврат названия блюда и калорий.
- Клиент: фото по камере или галерее → запрос к бэкенду → автозаполнение и сохранение.

## 4. Полная интеграция с LLM по каждому пункту

**Требование:** Основа приложения — человек может полностью делать всё в приложении через LLM (по каждому пункту).

### Реализация

| Область | Как завязать на LLM |
|--------|----------------------|
| **Задачи (План)** | Голос/текст → LLM извлекает задачу (название, дата/время, заметки). Советы по списку задач («что сделать первым», «перенести на завтра») от LLM. Редактирование задачи через диалог: «Перенеси встречу на 15:00» → LLM возвращает структурированный ответ, приложение обновляет задачу. |
| **Календарь** | «Какие встречи сегодня?» — ответ LLM на основе списка событий. «Добавь встречу с Иваном завтра в 14:00» — LLM извлекает параметры, бэкенд создаёт событие в Google Calendar через API. |
| **Почта** | «Прочитай последние письма» — краткое изложение от LLM. «Напиши ответ: согласен на встречу» — LLM генерирует текст письма, пользователь отправляет. «Удали спам» — LLM помечает кандидатов по списку писем или пользователь подтверждает. |
| **Здоровье** | Фото блюда → LLM (или отдельная модель) даёт калории и название. Советы по питанию/сну на основе дневника (калории, сон, активность) — один запрос к LLM с контекстом. |
| **Аналитика** | «Глубокий анализ» уже есть; расширить: передавать в LLM задачи, календарь, почту (кратко), здоровье — один общий промпт «Дай персональные рекомендации на день» и показывать ответ в приложении. |

Единая точка: один выбранный провайдер LLM (по умолчанию Gemini). Все запросы (извлечение задачи, совет, анализ, почта, календарь) идут через бэкенд к выбранной модели; на клиенте только формирование контекста и отображение ответа.

## 5. Удаление задач: свайп, настройки, контекстное меню

**Требование:** Удаление задач через свайп и через настройки; при долгом нажатии на задачу — меню (редактирование, удаление и т.д.).

### Реализация

| Элемент | Реализация |
|--------|-------------|
| **Свайп по задаче** | В списке задач использовать `List` + `.swipeActions(edge: .trailing)` с кнопкой «Удалить»; по действию вызывать `store.remove(task:)` и отмену уведомления. |
| **Долгое нажатие (контекстное меню)** | На строку задачи повесить `.contextMenu { ... }`: пункты «Редактировать» (открыть sheet редактирования/переноса), «Удалить» (с подтверждением при желании). Дополнительно: «Отложить» (то же, что текущий Reschedule). |
| **Настройки** | Отдельный экран/вкладка «Настройки»: блок «Задачи» — кнопки «Удалить выполненные», «Удалить все» (с диалогом подтверждения). После подтверждения — вызов `store.removeCompleted()` / `store.removeAll()` и отмена соответствующих уведомлений. |

---

## 5.1. Меню при долгом нажатии на задачу

**Пункты меню:** Редактирование (дата/время и при необходимости название/заметки), Удаление, при необходимости «Отложить» (отдельная кнопка или тот же редактирование).

## 6. Полная интеграция с нейросетью по выбору пользователя, по умолчанию Gemini

**Требование:** Пользователь выбирает нейросеть; по умолчанию — Gemini. Полная интеграция всех сценариев с выбранной моделью.

### Реализация

| Элемент | Реализация |
|--------|-------------|
| **Модели в приложении** | В перечислении моделей: **Gemini** (по умолчанию), Эвристика (офлайн), при желании OpenAI/другие. Сохранение выбора в UserDefaults (уже есть `aiModelKey`). |
| **Экран настроек** | Раздел «Модель ИИ»: Picker (или список) с вариантами «Gemini», «Эвристика», «OpenAI» и т.д. По умолчанию при первом запуске — Gemini. |
| **Бэкенд** | Единый эндпоинт или набор эндпоинтов для LLM: например `POST /llm/chat` с телом `{ "model": "gemini", "messages": [...] }` или отдельные маршруты для задач/календаря/почты/аналитики, но под капотом везде вызов выбранной модели (Gemini API ключ на бэкенде). Для анализа фото блюда — Gemini multimodal в `/analyze-meal`. |
| **Клиент** | Все запросы к «мозгу» (извлечение задачи, советы, глубокий анализ, в перспективе почта/календарь) отправляются на бэкенд с указанием или без (бэкенд использует сохранённую/дефолтную модель). Выбор модели в настройках меняет только отправку `model` в запросе или настройку на бэкенде по сессии. |

### Итог

- По умолчанию: **Gemini** (в коде и в UI).
- В настройках: явный выбор модели (Gemini, Эвристика, др.).
- Бэкенд: интеграция с Google AI (Gemini) для всех сценариев (задачи, анализ, почта, календарь, аналитика); при выборе «Эвристика» — можно не вызывать API, а считать на клиенте/бэкенде правилами.
