# План до готового приложения: что сделать, чтобы пользоваться постоянно

Ниже — все шаги по порядку. Выполнение по фазам позволит довести приложение до состояния «постоянно пользуюсь».

---

## Фаза 0. Что уже есть (база)

- **План:** задачи, голос, AI-разбор (эвристика + Gemini-заглушка), напоминания, свайп/меню удаление, настройки.
- **Аналитика:** статистика по дням, совет, «глубокий анализ» (бэкенд-правила).
- **Здоровье:** приёмы пищи, сон, активность, калории за день; фото → бэкенд (пока эвристика по размеру).
- **Календарь / Почта:** UI готов, но бэкенд не отдаёт реальные данные и не реализована авторизация Google.
- **Бэкенд:** только mock календаря, эвристика калорий, правила для LLM; нет OAuth, нет Google API, нет Gemini.

---

## Фаза 1. Бэкенд: авторизация Google (OAuth)

Без этого вкладки «Календарь» и «Почта» не работают с реальными данными.

### Шаг 1.1. Эндпоинт «Старт OAuth»

- В `jarvis-backend` добавить маршрут **GET /auth/google**.
- По запросу: сформировать URL авторизации Google OAuth 2.0 (client_id, redirect_uri, scope, state).
- **Scopes:** минимум  
  `https://www.googleapis.com/auth/calendar.events`,  
  `https://www.googleapis.com/auth/gmail.readonly`,  
  `https://www.googleapis.com/auth/gmail.modify`,  
  `https://www.googleapis.com/auth/gmail.compose`.
- Ответ: редирект (302) на этот URL (или вернуть JSON с `auth_url` для открытия в браузере с клиента).
- Использовать `credentials.json` (client_id, client_secret); redirect_uri должен совпадать с тем, что указан в Google Cloud Console (например `http://jarvis-app.mooo.com:8000/auth/callback`).

### Шаг 1.2. Эндпоинт «Callback» и сохранение токенов

- Добавить **GET /auth/callback**: query-параметры `code`, `state`.
- Обменять `code` на access_token и refresh_token (POST на Google token_uri).
- Сохранить токены. Варианты:
  - **A)** В памяти (dict по session_id из state) — для одного пользователя/устройства достаточно.
  - **B)** В БД (SQLite/файл) с привязкой к устройству (device_id в state).
- Установить cookie с session_id (или вернуть страницу «Успешно, закройте окно» и сохранить session в заголовке/query для мобильного клиента).
- Ответ: HTML-страница «Авторизация успешна. Можно закрыть вкладку» или редирект в custom URL scheme приложения (например `jarvis://auth/success?session=...`).

### Шаг 1.3. Эндпоинт «Проверка авторизации»

- **GET /auth/status**  
  Определять пользователя по cookie/session (или заголовку `X-Session-ID` / `Authorization`, если клиент передаёт идентификатор).
- Если токены есть и не истекли (или успешно обновлены через refresh_token) → ответ `{"authorized": true}`.
- Иначе → `{"authorized": false}`.
- При истёкшем access_token — обновлять его через refresh_token и затем возвращать true.

### Шаг 1.4. Клиент: передача сессии после OAuth

- После открытия `/auth/google` пользователь логинится в браузере и попадает на `/auth/callback`.
- Бэкенд отдаёт страницу с редиректом или ссылкой вида `jarvis://auth/success?session=SESSION_ID`.
- В приложении зарегистрировать URL scheme `jarvis` (Info.plist: CFBundleURLTypes) и в Scene/App обработать входящий URL: сохранить session_id в UserDefaults и считать пользователя авторизованным для запросов к календарю/почте.
- Все запросы к `/calendar/events` и `/mail/*` отправлять с cookie или заголовком (например `X-Session-ID: SESSION_ID`), чтобы бэкенд понимал, чьи токены использовать.

Итог фазы 1: пользователь нажимает «Войти через Google» в приложении → браузер → после успешного входа приложение получает сессию и при запросах к календарю/почте бэкенд использует сохранённые токены.

## Фаза 2. Бэкенд: Google Calendar API

### Шаг 2.1. Получение событий из Google Calendar

- В бэкенде по текущей сессии достать access_token (при необходимости обновить через refresh_token).
- Вызвать Google Calendar API: `calendar.events().list(calendarId='primary', timeMin=..., timeMax=..., singleEvents=True, orderBy='startTime')`.
- timeMin/timeMax — например «сейчас минус 1 день» и «сейчас плюс 30 дней».
- Преобразовать ответ в тот же JSON, что уже ожидает клиент: `id`, `title`, `notes`, `startDate` (ISO8601).
- **GET /calendar/events** возвращает этот список (вместо mock).

### Шаг 2.2. (Опционально) Лайв-обновления

- **Вариант A:** Клиент при открытии вкладки «Календарь» и по pull-to-refresh запрашивает GET /calendar/events — бэкенд каждый раз ходит в Calendar API. Для «постоянного пользования» часто достаточно.
- **Вариант B:** На бэкенде по таймеру (каждые 1–2 мин) опрашивать Calendar API и кэшировать; клиент запрашивает кэш.
- **Вариант C:** Google Calendar Push (webhook) — регистрация канала, при изменении календаря Google шлёт запрос на ваш URL; обновлять кэш и при следующем запросе клиента отдавать свежие данные.

Итог фазы 2: в приложении во вкладке «Календарь» отображаются реальные события из Google Calendar.

## Фаза 3. Бэкенд: Gmail API

### Шаг 3.1. Список писем

- **GET /mail/messages?max_results=15**  
  По сессии взять access_token, вызвать Gmail API: `users().messages().list(userId='me', maxResults=15)`.
- Для каждого messageId вызвать `messages().get()` и взять snippet, subject, from, date (из headers).
- Вернуть JSON-массив в формате, который уже ожидает клиент: `id`, `subject`, `from`, `date`, `snippet`.

### Шаг 3.2. Просмотр одного письма

- **GET /mail/messages/{id}**  
  Gmail API `messages().get(userId='me', id=id, format='full')`.
- Декодировать body (base64url), извлечь text/plain или text/html, вернуть JSON: `{ "id", "subject", "from", "date", "body": "..." }`.

### Шаг 3.3. Отправка письма

- **POST /mail/messages/send**  
  Тело: `{ "to": "...", "subject": "...", "body": "..." }`.
- Сформировать MIME-письмо (raw, base64url), вызвать `users().messages().send(userId='me', body=...)`.

### Шаг 3.4. Удаление письма

- **DELETE /mail/messages/{id}**  
  Вызвать Gmail API `messages().trash(userId='me', id=id)` (или `delete()` для полного удаления).

Итог фазы 3: в приложении можно просматривать письма, отправлять новые и удалять — при условии что клиент будет вызывать новые эндпоинты (см. фазу 7).

## Фаза 4. Бэкенд: Gemini (калории по фото и LLM)

### Шаг 4.1. API-ключ Gemini

- Получить API-ключ в Google AI Studio (aistudio.google.com).
- Хранить в переменной окружения на сервере (например `GEMINI_API_KEY`), не коммитить в репозиторий.

### Шаг 4.2. Анализ фото блюда через Gemini

- В **POST /analyze-meal**: принять тело запроса как изображение (binary или base64).
- Отправить изображение в Gemini (multimodal) с промптом вида: «What dish is shown? Reply with a short name and estimated calories for one typical portion. Format: title: ... ; calories: number».
- Распарсить ответ и вернуть `{"title": "...", "calories": N}`. При ошибке — fallback на текущую эвристику или `{"title": "Блюдо", "calories": 0}`.

### Шаг 4.3. LLM-советы (глубокий анализ)

- В **POST /llm/plan**: вместо фиксированных правил вызвать Gemini с контекстом: список задач (название, дата, выполнено/нет).
- Промпт: «По списку задач пользователя дай краткий персональный совет на день (1–3 предложения).»
- Ответ Gemini вернуть в поле `advice`.

Итог фазы 4: фото блюда даёт реальную оценку калорий; «Глубокий анализ» в Аналитике даёт текст от Gemini.

## Фаза 5. Бэкенд: единый чат/LLM для всех сценариев (расширение)

Чтобы «всё делать через LLM» (разбор задачи из голоса, советы по календарю/почте и т.д.):

### Шаг 5.1. Универсальный эндпоинт LLM

- **POST /llm/chat** (или несколько специализированных): тело `{ "model": "gemini", "messages": [ {"role": "user", "content": "..."} ] }` или контекст + запрос.
- Бэкенд вызывает Gemini API (chat или generateContent), возвращает ответ в едином формате.

### Шаг 5.2. Специализированные вызовы (по желанию)

- Разбор задачи из текста: эндпоинт принимает фразу, LLM возвращает структурированный JSON (title, date, notes) — клиент подставляет в форму задачи.
- Краткое изложение писем, генерация ответа на письмо и т.д. — все через вызовы к Gemini с соответствующими промптами и контекстом.

Итог фазы 5: приложение может использовать LLM для задач, календаря, почты, здоровья и аналитики; клиент остаётся тем же, меняются только запросы к бэкенду.

## Фаза 6. Клиент: обработка ошибок и обратная связь

Чтобы приложение было пригодно для постоянного использования, пользователь должен понимать, что пошло не так.

### Шаг 6.1. Синхронизация календаря на экране «План»

- Сейчас при ошибке синхронизации только `print(...)`. Добавить состояние ошибки (например `@State private var syncError: String?`) и показывать алерт или текст под кнопкой: «Не удалось загрузить календарь. Проверьте интернет и авторизацию.»

### Шаг 6.2. Здоровье: анализ фото

- Вместо `print("Nutrition analysis failed")` показывать пользователю сообщение: алерт или inline-текст «Не удалось распознать блюдо. Попробуйте ещё раз или введите вручную.»

### Шаг 6.3. Аналитика: глубокий анализ

- При ошибке запроса к `/llm/plan` показывать текст ошибки в блоке «Глубокий анализ» (например «Ошибка сети. Нажмите «Обновить» позже.»), а не оставлять пусто.

### Шаг 6.4. Общие сетевые ошибки

- При отсутствии интернета или 5xx от бэкенда не падать молча: показывать короткое сообщение и кнопку «Повторить» там, где это уместно (Календарь, Почта, Аналитика, Здоровье).

Итог фазы 6: при сбоях сети или бэкенда пользователь видит понятное сообщение и может повторить действие.

## Фаза 7. Клиент: почта — просмотр, написание, удаление

### Шаг 7.1. Просмотр письма

- При тапе на письмо в списке открывать экран просмотра (NavigationLink или sheet).
- Запрос **GET /mail/messages/{id}** для получения тела письма.
- Отобразить: тема, отправитель, дата, тело (текст или упрощённый HTML).

### Шаг 7.2. Кнопка «Ответить»

- На экране просмотра кнопка «Ответить» открывает экран написания письма с предзаполненными полями: Кому = from, Тема = «Re: …», тело можно оставить пустым или подставить цитату.

### Шаг 7.3. Экран «Новое письмо»

- Поля: Кому, Тема, Текст. Кнопка «Отправить» → **POST /mail/messages/send** с телом `{ to, subject, body }`.
- После успешной отправки закрыть экран и при желании обновить список.

### Шаг 7.4. Удаление письма

- В списке писем: свайп по строке → кнопка «Удалить» → вызов **DELETE /mail/messages/{id}** и удаление из локального списка.
- На экране просмотра: кнопка «Удалить» с подтверждением → тот же DELETE.

Итог фазы 7: в приложении можно полноценно читать почту, отвечать, писать новые письма и удалять.

## Фаза 8. Клиент: камера для фото блюда (опционально, но удобно)

- На экране «Здоровье» в блоке «Питание» добавить кнопку «Сфотографировать» (иконка камеры).
- Использовать `UIImagePickerController` (или `CameraView` в SwiftUI) для съёмки фото.
- Отправить полученное изображение на **POST /analyze-meal** так же, как при выборе из галереи, и подставить результат в форму.

Итог: можно сфотографировать блюдо прямо в приложении, не только выбрать из галереи.

## Фаза 9. Развёртывание и конфигурация

### Шаг 9.1. Бэкенд на сервере

- Убедиться, что бэкенд запущен по адресу, который указан в приложении (например `http://jarvis-app.mooo.com:8000`).
- В Google Cloud Console для OAuth redirect URI указан именно этот хост и путь `/auth/callback`.
- Переменные окружения: `GEMINI_API_KEY`, путь к `credentials.json` (или переменные для client_id/client_secret).

### Шаг 9.2. HTTPS (рекомендуется)

- Сейчас в приложении допущен HTTP (ATS exception). Для постоянного использования лучше поднять HTTPS (nginx + Let's Encrypt) и в Config указать `https://...`, убрать исключение ATS для продакшена.

### Шаг 9.3. Один конфиг для разработки и продакшена

- В Xcode: разные схемы или Build Configuration с разным `backendURL` (например Debug → localhost или тестовый сервер, Release → production URL), чтобы не менять код при сборке под себя.

Итог фазы 9: бэкенд стабильно доступен, приложение указывает на правильный URL, при желании — через HTTPS.

## Фаза 10. Полировка перед «постоянным использованием»

### Шаг 10.1. Уведомления

- Проверить, что напоминания по задачам реально приходят в нужное время (тест на устройстве).
- При удалении или переносе задачи напоминание отменяется/обновляется (уже реализовано).

### Шаг 10.2. Сохранение данных

- Данные задач и здоровья хранятся в UserDefaults — при переустановке приложения они теряются. Для «постоянного использования» можно позже добавить экспорт/импорт (JSON или файл) или синхронизацию через бэкенд; на первый релиз не обязательно.

### Шаг 10.3. Календарь на iPhone (EventKit)

- По ТЗ события из календаря iPhone тоже должны быть в приложении. Реализация: на клиенте запросить доступ к EventKit, читать события из выбранных календарей за период и объединять с событиями из GET /calendar/events в одном списке с пометкой источника (Google / iPhone).

### Шаг 10.4. Иконка и экран запуска

- Убедиться, что в Assets есть иконка приложения для всех нужных размеров.
- При необходимости добавить Launch Screen (Storyboard или SwiftUI), чтобы не было пустого белого экрана при старте.

Итог фазы 10: приложение ведёт себя предсказуемо, напоминания работают, при желании — локальный календарь и внешний вид в порядке.

## Минимальный путь «могу пользоваться постоянно»

Если нужно быстрее выйти на «постоянное использование» без всего сразу:

1. **Обязательно:** Фаза 1 (OAuth) + Фаза 2 (Calendar API) — тогда календарь в приложении живой.
2. **Обязательно:** Фаза 4 (Gemini для калорий и llm/plan) — тогда фото и аналитика полезны.
3. **Обязательно:** Фаза 6 (ошибки в UI) — тогда понятно, когда что-то не работает.
4. **Желательно:** Фаза 3 + Фаза 7 — тогда и почтой можно пользоваться в приложении.
5. **По возможности:** Фаза 9 (HTTPS и конфиг), Фаза 10 (напоминания, иконка).

После этого приложение уже можно использовать постоянно: задачи, календарь, здоровье, аналитика с LLM, при необходимости почта, с понятными ошибками и стабильным бэкендом.
